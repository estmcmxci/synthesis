Trust Resolution Layer

A primitive for verifiable agent identity, capability discovery, and version integrity — anchored to ENS.

Overview
Agents can't trust each other — there's no standard way to verify identity, discover capabilities, or detect silent changes between interactions. Ensemble is a CLI tool and Trust Resolution Layer that composes ENS standards (ENSIP-25, ENSIP-26), Agent Identity Profiles, and domain-verified SKILL.md files into a single deterministic verification flow. The ensemble CLI is the product — the infrastructure tool for setting records, signing manifests, verifying trust, pinning content, and launching identity tokens. Given any ENS name, resolve the full trust stack in one pass: personhood (World ID), identity (ERC-8004), context, signed manifest, and skill provenance. The output is a structured TrustProfile with a progressive trust tier — from 'none' to 'full' — that agents use to decide whether to cooperate, delegate, or refuse. The personal site at emilemarcelagustin.eth is the reference implementation — a visual wrapper around the CLI that makes the resolution output tangible and interactive.

Product Layer — Identity Token

Wallet Layer — Agent Wallet
Identity to Action

The TRL says who an agent is. The wallet lets it spend. The wallet layer bridges identity and action through a Safe smart account on Base, owned by a biometric passkey. No seed phrase, no private key in a .env, no custodian. The passkey lives in the device's Secure Enclave — it cannot be exported, phished, or delegated. The human's body is the ultimate signer.

The agent never holds the owner key. Instead, it receives a scoped session key — a cryptographic capability that lets it sign transactions within policy bounds. The SmartSession module enforces policies on-chain: allowed contracts, spending limits, time windows. Exceed any bound and the transaction reverts.

Spending Layer — Agent Card + x402
Three Rails, One Identity

The spending layer closes the loop. A TRL-verified agent that can act in the world — paying for API calls, domain renewals, cloud bills — without a human copying card numbers into a form. The agent's on-chain USDC balance is the funding source. No separate bank account, no manual top-ups.

Bankr provides the execution layer for crypto operations. The Safe provides self-sovereign ownership via passkey. x402 adds trust-verified agent-to-agent payments — before paying, the agent resolves the payee through TRL; before responding, the payee resolves the payer. Both sides verify identity before money moves. The virtual card bridges to fiat.

Trust-Scoped Spending

The card issuer doesn't decide the agent's limits — the TRL does, enforced by the session key's on-chain policy. An agent with trustScore: "full" gets higher ceilings than one with "registered". Limits are enforced at three levels: on-chain smart contract escrow, card issuer API, and MCP policy configuration.

MCP-native. The agent requests card details and executes payments through MCP tools. No human in the loop for authorized transactions — the session key policy is the authorization. The whole system is verifiable: identity on-chain, policies on-chain, spend history on-chain.

Coinbase's x402 has no identity verification — you're paying an address with no way to verify who's behind it. TRL-gated x402 means both sides verify each other before money moves. No anonymous payments to unverified addresses. The identity layer makes spending safe, not just possible.
